# Running Ionosphere in Docker

Ionosphere is designed to run securely and efficiently within a lightweight Node.js Docker container. 

## Quick Start

1. **Clone and Configure**
   ```bash
   git clone https://github.com/h0tp-ftw/ionosphere.git
   cd ionosphere
   cp .env.example .env
   ```
2. **Setup your Environment**
   Edit the `.env` file to include your Google API key:
   ```env
   GEMINI_API_KEY=AIzaSy...
   ```
3. **Build and Run**
   ```bash
   docker-compose up -d --build
   ```

Ionosphere is now running on `http://localhost:3000`.

## Configuration & Persistence

### 1. The Settings File (`.gemini/settings.json`)
The `settings.json` file controls the Gemini CLI's core behavior (like model selection, generation configs, and global MCP servers). 

In Ionosphere, this file is automatically generated at startup based on your `.env` variables via the `scripts/generate_settings.js` script. You do not need to mount a static `settings.json` file into the container. Simply configure your `.env` file and the startup script will handle the rest.

### 2. Session Persistence (Stateful Mode)
If you are running Ionosphere with `SESSION_MODE=stateful`, you will want to persist the CLI's conversation history across container restarts.

To do this, mount the `~/.gemini/sessions/` directory to a volume or host path in your `docker-compose.yml`:

```yaml
services:
  ionosphere:
    build: .
    ports:
      - "3000:3000"
    volumes:
      - ./gemini_sessions:/root/.gemini/sessions  # Persist LLM conversations
      - ./temp_storage:/app/temp                  # Persist Routing DB
```

### 3. Dynamic MCP Servers
You do NOT need to hardcode your MCP servers into the Docker image's setup script.

Instead, you can inject them dynamically per-request by passing the `mcpServers` object in your POST payload to `/v1/chat/completions`. Ionosphere's isolated workspace engine will automatically scaffold a temporary environment with your servers just for that single execution turn. 

Refer to `API.md` for request syntax.

## Building Specific CLI Versions

You can control which version of the underlying `@google/gemini-cli` is installed during the image build process by using the `GEMINI_CLI_TAG` build argument:

```bash
docker build --build-arg GEMINI_CLI_TAG=preview -t ionosphere .
```
This is useful if you want to test alpha/beta features of the official CLI that have not yet reached the `latest` npm tag.
